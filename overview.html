What is being built

Seven layers, one system

Every layer serves a specific purpose. Each one must be built, tested, and proven before the next begins. Tap any layer to understand what it does.

01 Physical sensors
The hardware
Pressure transducers, flow meters, RTDs, torque flanges, and encoders wired to the bench. These convert physical quantities into electrical signals — 4–20 mA current loops or 0–10 V voltage — that the software can read. Every measurement the system makes starts here. If a sensor is wrong, everything downstream is wrong.
02 Data acquisition & scaling
Raw → engineering units
DAQ hardware digitises the analog signals at 100–1000 samples per second. The software then applies calibrated scaling to convert raw voltages into meaningful engineering values — bar, L/min, °C, RPM, Nm. This layer is where sensor calibration certificates are applied and where a broken wire (0 mA on a 4–20 mA loop) is detected as a fault.
03 Safety monitor
Runs independently
The most critical layer. It runs in its own dedicated thread, completely independent of the test sequence. Every sensor reading is checked against multi-level alarm limits on every cycle. If a critical limit is exceeded, outputs go to zero immediately — this layer does not wait for the state machine. It also services the hardware watchdog, so a software crash cannot leave the bench pressurised.
04 State machine & PID control
Test sequence logic
The state machine orchestrates the test sequence — Idle → Initialise → Ramp Up → Hold → Ramp Down → Shutdown → Report — ensuring each phase completes correctly before the next begins. The PID controller continuously adjusts the proportional valve to maintain the target pressure or flow, correcting for system variations in real time. These two systems work together to make the test automatic, repeatable, and safe.
05 Operator interface
What the technician sees
The desktop HMI built in PyQt5 and PyQtGraph. Live gauges for every sensor channel, a scrolling trend chart, colour-coded alarm indicators, and the Start/Stop controls. The layout is designed around the technician's workflow — wireframed and reviewed before any code is written. A well-designed interface on a hydraulic bench is a safety feature, not an aesthetic one.
06 Data logging
The record
Runs asynchronously — it never blocks the control loop. Every sensor reading is timestamped and written to CSV and a SQLite database throughout the entire test. Fault events get a full sensor snapshot at the moment they fire. Log files are write-once. Nothing is lost, nothing can be altered after the fact.
07 Reporting & test certificate
The deliverable
On test completion, the software automatically generates a professional PDF certificate containing the component details, test conditions, all sensor results, pass/fail assessment against specification limits, trend graphs, and calibration references. The test register database is updated. The certificate is ready for customer delivery without a technician touching a spreadsheet.

Safety architecture

Built to be safe first

Safety is not a feature added at the end — it is a structural constraint baked into the architecture from the start. Three design decisions enforce this.

01 — Independent thread
Safety runs separately — always
The safety monitor runs in its own dedicated thread at the highest OS priority. A bug in the test sequence, a frozen state machine, or a slow UI cannot prevent safety from firing. If a sensor exceeds a critical limit, outputs go to zero immediately — on that cycle. The watchdog timer ensures a software crash cannot leave the bench pressurised.
02 — E-stop integration
Hardware and software working together
The physical E-stop button on the bench is wired directly to the hardware relay — hardware safety is always primary. BenchVision also monitors the E-stop digital input on every safety cycle: pressing the button triggers an immediate software shutdown, fault logging with a full sensor snapshot, and a controlled sequence that closes valves and depressurises safely. Both layers must be satisfied for the bench to run.
03 — Hardware-agnostic design
The safety layer does not depend on any one DAQ
The safety monitor reads from an abstracted sensor interface — the same interface used by the simulator. This means the entire safety logic can be developed, unit-tested, and validated before any physical hardware is connected. Swapping the DAQ board — or adding a new bench type — requires only a new driver implementation, not changes to the safety logic itself.
Responsibility boundary: BenchVision's safety layer is advisory software — it enforces configured limits and signals alarms. Hardware safety (pressure relief valves, physical interlocks, E-stop relay) remains the responsibility of the bench builder. Software limits are a second line of defence, not a substitute for hardware safety engineering.

Design principle

Modular by design, safe by default

Every module has a single responsibility. Each can be tested independently. The architecture is designed to adapt to any bench configuration — this installation is the first of many.

01 — Acquisition
Sensor-agnostic reader
The acquisition module works with any DAQ hardware or protocol — NI-DAQmx, Modbus, serial. Swap the driver, keep everything else. The simulator and real hardware share the same interface.
02 — Safety
Independent thread
Safety logic runs in its own thread at the highest priority. A bug in the test sequence cannot prevent the safety monitor from firing. Hardware safety always comes first — software is the second line of defence.
03 — Sequences
YAML-defined tests
Test procedures are defined in configuration files, not hardcoded. Adding a new test type means creating a new YAML file — no code changes required. The state machine reads the definition and executes it.
04 — Reporting
Templated certificates
Certificate layout, company branding, and content are fully templated. Different bench types produce different certificates with no changes to the core logging or analysis code.

"Complete something every session. A working sensor simulator is more valuable than half a complete system. Modular build means you always know exactly where you are."
